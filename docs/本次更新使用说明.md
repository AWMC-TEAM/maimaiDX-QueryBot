# QueryBot 账号、BREAK 与管理台使用说明

## 1. 首次使用：用户协议

1. 发送 `用户协议`。
2. Bot 返回 AWMC maiBot 服务协议 v4：
   `https://wiki.awmc.team/guide/bot/terms`。
3. 打开链接并阅读完整内容。
4. 完整复制网页中的确认词，直接发给 Bot；也可发送
   `同意用户协议 <完整确认词>`。
5. 确认成功后才能使用账号绑定、上传、发票、BREAK 转账和抽奖。

发送 `撤回用户协议` 可撤回同意。管理员在 WebUI 修改确认词后，
系统会自动更新协议版本，旧用户需重新确认。

## 2. 直接发送 SGWCMAID 或二维码链接

支持以下三种输入，解析结果会统一为 `SGWCMAID...`：

- `SGWCMAID...` 完整字符串；
- `https://wq.wahlap.net/qrcode/img/MAID....png` 图片链接；
- `https://wq.wahlap.net/qrcode/req/MAID....html` 请求链接（允许携带查询参数）。

当一条消息直接以上述任一内容开头时，执行顺序为：

1. Bot 以最高优先级尝试撤回原消息。
2. 检查用户协议和群查询功能开关。
3. 拉取并保存 PC 数据。
4. 读取用户的上传 Token：
   - 只绑定水鱼：只上传水鱼；
   - 只绑定落雪：只上传落雪；
   - 两者都绑定：两者都上传；
   - 都没绑定：仅同步 PC。
5. 返回 PC 数量、上传结果、BREAK 结算与 REF_ID；不显示街机 UID。

如果 Bot 没有撤回权限，会明确提示用户立即手动撤回。

### 二维码失效与重试

- 无效、过期或 AWMC 暂时无法读取时，Bot 会请用户重新获取。
- 3 分钟内最多尝试 3 次，每次都会显示 `1/3`、`2/3`或 `3/3`。
- 失败的二维码不会被去重缓存拦截，可立即发送新凭据重试。
- 第 3 次仍失败时结束本轮流程，需稍后重新开始。
- 成功后自动清空重试计数。

## 3. 交互式绑定

发送 `mai绑定` 且不带参数时：

1. Bot 请求用户发送最新 SGWCMAID。
2. 用户发送凭据后，Bot 先尝试撤回，再校验账号。
3. 格式错误、二维码过期或校验失败时，允许最多 3 次输入。
4. 发送 `取消`、`cancel`、`q` 或 `退出` 可结束。
5. 绑定成功后只显示玩家名和 Rating，不显示街机 UID。

也可一次发送 `mai绑定 SGWCMAID...`，但原消息仍会尝试撤回。

别名：`maibind` = `mai绑定`，`maibindfish` = `mai绑定水鱼`，
`maibindlx` = `mai绑定落雪`。如果最新 SGWCMAID 验证出的舞萌 UID 已存在于
迁移旧记录，Bot 会自动认领并安全转移旧配置，旧记录的舞萌/PC 凭据同时失效。
认领不转移 BREAK、协议/封禁状态或落雪 OAuth；OAuth 需当前用户本人执行 `lxbind`。

### maiu / maiul / maiua 重试

- `maiu`：上传水鱼；`maiul`：上传落雪；`maiua`：同时上传两者。
- 落雪优先使用 `lxbind` OAuth 直接上传，无需导入 Token；旧导入 Token 仅作兼容备用。
- 无参数时先使用已绑定二维码；也可直接发送 `maiu SGWCMAID...`。
- 如果返回 HTTP 500、全部 `null`、登录失败或二维码无效/过期，Bot 会请求补交最新二维码。
- 最多尝试 3 次，每次补交的凭据都会尝试撤回；发送 `取消` 可退出。
- 第 3 次仍失败时结束流程。所有失败尝试均不扣 BREAK。

## 4. BREAK 默认规则

| 业务 | 默认值 | 免费/结算规则 |
|---|---:|---|
| 签到基础 | 1～2 | 连续签到最高额外 +2 |
| 猜歌 | +1 | 每次猜对固定给，无每日上限 |
| 查分器 API | 1 | 每日首次实际 API 请求免费，缓存命中不扣 |
| 分析 b50 | 3 | 无首免；高峰时段按原规则可加倍 |
| 上传水鱼 | 2 | 所有上传方式共享每日首次成功免费 |
| 上传落雪 | 2 | 所有上传方式共享每日首次成功免费 |
| 同时上传 | 3 | 所有上传方式共享每日首次成功免费 |
| 发票 | 倍率 × 2 | 发票每日首次成功免费 |
| 转账手续费 | 0 | 可在后台调整 |
| 抽奖 | 2/次 | 单次可抽 1～10 次 |

发票例子：2 倍票 4 BREAK，3 倍票 6 BREAK，4 倍票 8 BREAK，
5 倍票 10 BREAK，6 倍票 12 BREAK。

外部业务仅在真正成功后结算。HTTP 200 但业务返回失败、异步任务失败或超时
都不扣 BREAK。公共网关异步上传会轮询到完成后再结算。

## 5. BREAK 互动

- `转账BREAK @用户 10`：转账 10 BREAK。
- `BREAK抽奖`：抽 1 次。
- `BREAK抽奖 5`：连抽 5 次。
- `我的AWMC`：查看余额、签到、使用量和最近流水。

## 6. 吃分推荐

`今日吃分推荐` 会将成功结果渲染为图片，避免长文本刷屏。
缺少昨日存档、无全量成绩等错误仍使用短文字提示。

## 7. 精简消息模式

默认配置 `MAIMAIDX_COMPACT_MESSAGES=true`。开启后会合并猜歌开场与结算、群排行摘要、
把凭据撤回失败警告附在最终结果或重试提示中，并省略不必要的“正在处理”消息。
二维码三次重试、猜歌阶段线索等需要继续交互的内容仍会正常发送。若更希望所有耗时
操作先发送进度提示，可将该配置设为 `false` 并重启 Bot。

## 8. 管理 WebUI

详细部署、反向代理、配置项和故障排查见 [WebUI 配置说明](WebUI配置说明.md)。

启用后可管理：

- 用户账号摘要、BREAK 余额、封禁/解封；
- 每一笔 BREAK 流水，包括首次免费的 0 扣费记录；
- 签到、猜歌、上传、发票、转账和抽奖参数；
- REF_ID 完整处理链路、外部请求、异常与耗时；
- 用户协议链接、网页确认词和版本；
- 群功能开关、消息排行和命令调用报表。

后台不返回二维码、Token 或完整街机 UID。所有管理修改也会生成 REF_ID。

## 9. Koishi 数据库迁移

可直接使用包含其它插件表的完整 Koishi SQLite 数据库。迁移器以只读方式
打开源库，只迁移 `maibot_bindings` 和 `maibot_user_terms`，其它表保持不变。

### Bot 管理指令（推荐）

1. 把 Koishi 数据库复制到 `MAIMAIDX_KOISHI_MIGRATION_DIR`，默认是插件目录下的
   `data/migration`。
2. 超级管理员先发送 `迁移Koishi 检查 koishi.db`。
3. 核对预检统计后发送 `迁移Koishi 确认 koishi.db`。

迁移器会尝试通过 Koishi `binding` 表的 `aid/pid` 自动解析 QQ 身份。如果仍有
`koishi:<id>` 无法映射，把以下格式的 `identity-map.json` 放在同一目录：

```json
{"koishi:12": "123456789"}
```

然后执行：

```text
迁移Koishi 检查 koishi.db identity-map.json
迁移Koishi 确认 koishi.db identity-map.json
```

该指令限制为超级管理员，且拒绝读取配置目录以外的路径。预检不会写入源库或目标库；
确认迁移使用可重复执行的 upsert，不会清空 QueryBot 已有记录。

### 命令行方式

```bash
python3 scripts/migrate_maibot_accounts.py /path/to/koishi.db --dry-run
python3 scripts/migrate_maibot_accounts.py /path/to/koishi.db
```

迁移器会优先自动读取 Koishi `binding` 表；无法自动解析时可附加身份映射 JSON：

```bash
python3 scripts/migrate_maibot_accounts.py /path/to/koishi.db \
  --identity-map /path/to/identity-map.json
```

旧协议记录会迁入审计库，但不会替代 v4 协议确认；用户仍需重新阅读并发送网页确认词。

## 10. SQLite / YAML / MySQL 存储

支持 `MAIMAIDX_STORAGE_BACKEND=sqlite|yaml|mysql`。管理员先使用
`存储迁移 检查 <源> <目标>` 预检，再使用 `存储迁移 确认 <源> <目标>` 执行。
完整配置、MySQL 建库权限、回滚与错误说明见
[统一存储与迁移说明](统一存储与迁移说明.md)。今日新增的全部环境变量见
[今日新增 ENV 配置](今日新增ENV配置.md)。
