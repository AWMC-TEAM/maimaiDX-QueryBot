# QueryBot 统一存储与无损迁移

## 后端模型

- `sqlite`：沿用当前各模块的 SQLite 与 JSON 文件，默认值。
- `yaml`：将持久状态写入一个带逐文件 SHA-256 的 YAML 快照。
- `mysql`：将完整快照事务写入 MySQL，校验完成后才切换活动快照。

为兼容现有模块，YAML/MySQL 模式保留本地工作缓存，所选后端作为持久化源。Bot 启动时
恢复，运行中定时同步，正常退出时再同步一次。音频缓存、迁移源文件和 `.DS_Store` 不进入
快照；账号、Token、BREAK、审计、封禁、PC、落雪、白名单、用户成绩快照以及群功能和
猜歌状态都会迁移。

> 迁移完成后不要删除本地 `data`。当前 MySQL 模式不是把每一条业务 SQL 直接改写到
> MySQL，而是“MySQL 持久化快照 + 本地 SQLite/JSON 工作集”。本地文件负责低延迟运行，
> MySQL 负责校验、恢复和备份。直接删除工作集会让已经打开的 SQLite 模块缺表或丢状态。

## MySQL 配置

```sql
CREATE DATABASE maimaidx CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'maimaidx'@'%' IDENTIFIED BY 'replace-with-strong-password';
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, INDEX, REFERENCES
  ON maimaidx.* TO 'maimaidx'@'%';
FLUSH PRIVILEGES;
```

```env
MAIMAIDX_STORAGE_BACKEND=sqlite
MAIMAIDX_STORAGE_NAMESPACE=production
MAIMAIDX_STORAGE_MYSQL_HOST=127.0.0.1
MAIMAIDX_STORAGE_MYSQL_PORT=3306
MAIMAIDX_STORAGE_MYSQL_USER=maimaidx
MAIMAIDX_STORAGE_MYSQL_PASSWORD=replace-with-strong-password
MAIMAIDX_STORAGE_MYSQL_DATABASE=maimaidx
MAIMAIDX_STORAGE_MYSQL_CHARSET=utf8mb4
MAIMAIDX_STORAGE_MYSQL_TABLE_PREFIX=maimaidx_
MAIMAIDX_STORAGE_MYSQL_SSL=false
MAIMAIDX_STORAGE_MYSQL_KEEP_SNAPSHOTS=3
MAIMAIDX_STORAGE_SYNC_INTERVAL_SECONDS=900
```

迁移完成前保持 `MAIMAIDX_STORAGE_BACKEND=sqlite`。

## 管理员迁移指令

```text
存储状态
存储迁移 检查 sqlite mysql
存储迁移 确认 sqlite mysql
```

确认成功后设置 `MAIMAIDX_STORAGE_BACKEND=mysql` 并重启。其它方向同样支持：

```text
存储迁移 检查 mysql yaml
存储迁移 确认 mysql yaml
存储迁移 检查 yaml sqlite
存储迁移 确认 yaml sqlite
```

迁移到 SQLite 时先暂存恢复任务，重启时才校验并应用，避免覆盖运行中已打开的数据库。
`存储同步` 可立即刷新当前 YAML/MySQL 后端。

## 低资源模式

定时任务先检查受管文件的路径、大小、纳秒修改时间和 SQLite WAL；工作集没有变化时，
不会读取数据库内容、制作快照或连接 MySQL。制作快照时会在目录遍历阶段直接剪掉音频、
迁移文件以及配置为不持久化的成绩缓存目录，避免扫描数 GiB 的可重建缓存。MySQL 写入后
使用流式逐文件校验，不再把完整快照额外复制一份到内存。

推荐大型部署使用：

```env
MAIMAIDX_STORAGE_BACKEND=mysql
MAIMAIDX_STORAGE_SYNC_INTERVAL_SECONDS=900
MAIMAIDX_STORAGE_INCLUDE_USER_SCORES=false
MAIMAIDX_STORAGE_INCLUDE_PLAYER_CACHE=false
MAIMAIDX_STORAGE_BOOTSTRAP_POLICY=auto
```

`900` 表示每 15 分钟检查一次变更；`0` 表示关闭周期同步，仅在启动、正常关机和管理员
发送 `存储同步` 时同步。设置为 `0` 最省资源，但机器突然损坏时，MySQL 可能缺少上次
手动或正常关机后的新增数据。`auto` 会沿用已经与 MySQL 关联的本地工作集，避免每次启动
都从远端下载完整快照。

## 无损与失败处理

- SQLite 通过在线 backup API 读取一致快照。
- YAML 先写临时文件，校验后原子替换，并保留 `.bak`。
- MySQL 在事务内写入新快照；失败回滚，不切换活动快照。
- 数据未变化时不会重复写 MySQL；默认仅保留最近 3 个成功快照。
- 恢复前校验总体 manifest 和逐文件 SHA-256；替换失败恢复备份。
- “检查”只验证源快照和目标配置，“确认”才写入。

错误会直接说明原因，例如 MySQL 地址配置缺失、连接被拒绝、YAML 文件不存在、目录不可写、
快照校验失败或后端名称错误。默认 `MAIMAIDX_STORAGE_FAIL_FAST=true`：选择 YAML/MySQL
但初始化失败时拒绝启动，防止悄悄使用过期本地数据。

空 YAML/MySQL 默认也拒绝直接启用，必须先迁移。只有确定要用现有本地数据初始化一个
全新后端时，才临时设置 `MAIMAIDX_STORAGE_ALLOW_EMPTY_REMOTE_INIT=true`。

历史成绩和最新玩家缓存可能达到数 GiB。默认两者都会进入快照；大型部署可设置
`MAIMAIDX_STORAGE_INCLUDE_USER_SCORES=false` 和
`MAIMAIDX_STORAGE_INCLUDE_PLAYER_CACHE=false`，将其保留为本地可重建缓存，并使用独立
namespace 保存迁移时生成的完整历史归档，避免每次小变更都重写数 GiB 数据。

YAML 快照包含二维码与上传 Token，程序会尽量设置为仅当前系统用户可读（`0600`）；
仍应放在受保护目录，禁止提交到 Git 或由 Web 服务器公开。
